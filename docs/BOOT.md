ClaudeCode向け 要件プロンプト（macOS画像ビューワー / Swift）

あなたは macOS 向けの画像ビューワーアプリを Swift（AppKit優先、必要なら一部SwiftUI） で実装してください。目的は「フォルダ内の大量画像を、待ち時間なく次々に確認し、削除・情報確認・仕分けを高速に行える」ことです。特に フォルダに1000〜2000枚あっても、フォルダを開いた直後の“最初の1枚”が即表示され、カーソル連打でもストレスがないことを重視します。

1. 対象プラットフォーム
   • macOS ネイティブアプリ
   • 実装言語：Swift
   • UIフレームワーク：AppKit を優先（パフォーマンスと仮想化重視）

⸻

2. UI/機能要件

2.1 メニュー要件
• フォルダを開く
• ユーザーが任意のフォルダを選択して、そのフォルダ内の画像を閲覧できる
• 最近使ったフォルダ
• 最近開いたフォルダの履歴を表示し、選択すると再度そのフォルダを開ける
• 履歴は永続化する（アプリ再起動後も残る）

2.2 表示要件
• 操作単位は 基本的にフォルダ単位
• フォルダを開いたら 最初の1枚を表示
• 「最初」の定義は実装側で妥当な初期値を設定（例：ファイル名ソートの先頭）
• ただし“最初の1枚の表示”が遅くならないようにする
• メイン表示：
• ウィンドウ内で可能な限り大きく画像を表示（アスペクト比維持、レターボックス可）
• 下部表示：
• 画面下部に カルーセル形式のサムネイル一覧
• 現在表示中の画像は 枠線でハイライト
• 大量画像でもスクロール/描画が重くならないように、表示領域の仮想化を行う

⸻

3. ショートカット要件（必須）

以下のキー操作を確実に実装してください（編集可能なキーバインド機能は不要、固定でよい）。

ショートカット一覧
• ← / →（カーソル左右）：前後に 1枚進む
• 連打してもUIが固まらないこと
• d：削除
• 削除対象は「現在表示中の画像」
• 削除後は自然な挙動で次の画像を表示（例：次、なければ前）
• i：情報を見る
• 画像から EXIFを解析し、**プロンプトデータ（例：生成AI関連メタデータ等）**を含む情報を表示
• 表示UIはモーダル/サイドパネル等、実装は任意。ただし操作を阻害しすぎないこと
• Space（スペース）：表示/非表示（常に）
• “常に”の意味：アプリ全体として、表示要素（下部カルーセルやUIオーバーレイ等）をトグルする想定
• 具体的に何を隠すかは実装提案してよいが、ユーザーが「画像だけに集中する」状態を作れること

⸻

4. パフォーマンス要件（最重要）

以下を満たす設計・実装にしてください。

4.1 最初の1枚を最速で表示
• フォルダを開いた直後、フォルダ内全画像のスキャン完了や全サムネ生成を待たずに、最初の1枚を即表示する
• フォルダ内の全メタデータ取得・サムネ生成は バックグラウンドで段階的に実施する

4.2 カーソル移動の待ちを無くす（先読み必須）
• カーソル左右での移動で、I/Oやデコード待ちが露骨に発生しないようにする
• 実装方針：
• 現在表示中の前後数枚を先読み（prefetch）
• 連打に備えて、先読みは「次方向を厚め」など優先度をつける
• 表示対象が変わったら、不要なバックグラウンド処理は キャンセルできる設計にする

4.3 サムネ生成は低優先度
• 下部カルーセルのサムネ生成は、メイン表示や先読みを阻害しない
• サムネは必要になった分から生成し、表示範囲外は後回しにする

⸻

5. 画像フォーマット・入出力の前提
   • フォルダ内に一般的な画像形式（例：JPEG/PNG/HEIC等）が混在し得る
   • macOSの標準的な画像デコード機構（ImageIO等）を利用してよい

⸻

6. 受け入れ基準（Acceptance Criteria）
   • フォルダを開くと、最初の1枚がすぐ表示される（全サムネ生成待ちが発生しない）
   • ←/→ の連打でも UI が固まらず、表示切り替えが追従する
   • 下部カルーセルが表示され、現在画像が枠線ハイライトされる
   • d で現在画像を削除でき、削除後に次の画像へ自然に遷移する
   • i でEXIF由来の情報（プロンプトデータ含む）を閲覧できる
   • Space で“表示/非表示”がトグルでき、画像だけの表示状態にできる
   • 「最近使ったフォルダ」メニューから再オープンできる

## アーキテクチャなど

1. 全体アーキテクチャ

   • UI層（AppKit View / Controller）と、ドメイン層（フォルダ/画像モデル）、サービス層（ローダー/キャッシュ/メタデータ）の分離方針
   • 状態管理（現在フォルダ、現在インデックス、表示モード、最近使ったフォルダ、選択状態など）

   2. 主要コンポーネント（Swift/AppKit）

   • 例：AppDelegate/Scene, MainWindowController, ImageViewerViewController, ThumbnailCarouselViewController(or NSCollectionView), InfoPanelController
   • モデル：FolderSession, ImageItem, RecentFolderStore, FavoritesStore
   • サービス：DirectoryScanner, ImageDecodeService, ThumbnailService, PrefetchController, CacheManager, ExifService
   • それぞれの責務・公開API・依存関係（矢印）を明確化

   3. 非同期処理設計

   • 使用する仕組み（例：Swift Concurrency Task / TaskGroup、または OperationQueue）の選定理由
   • 優先度付きキューの設計（P0=表示、P1=前後先読み、P2=さらに先、P3=サムネ生成）
   • UIスレッドでやること/やらないことの明確化

   4. サムネ生成

   • NSCollectionView の仮想化前提で「表示範囲のサムネから生成」する設計
   • サムネの メモリキャッシュ（LRU） と ディスクキャッシュ（永続） の方針
   • キャッシュキー（path + mtime + 生成サイズなど）と破棄戦略

   5. 先読み戦略（プリフェッチ）

   • 現在インデックス i を中心にした スライディングウィンドウ（例：前3/後12など）の初期値
   • 進行方向に応じて後方を厚くするなどの最適化案
   • フォルダ切り替え・高速連打・ジャンプ（サムネクリック）時の挙動

   6. キャンセル戦略（最重要）

   • カーソル連打やフォルダ切り替えで不要になったデコード/サムネ生成を即キャンセルする方法
   • 「表示対象が変わったら古いタスクを止める」実装パターン（例：キャンセルトークン、Taskの管理、世代IDなど）
   • キャンセルしてもキャッシュとして活かせるもの/捨てるものの区別

   7. エッジケースとテスト観点

   • 2000枚以上、巨大解像度、壊れた画像、権限、削除後のインデックス、最近使ったフォルダの永続化
   • 受け入れ基準に対応する簡易テスト戦略（手動/自動の粒度）

   8. 最初の1枚を最速表示する具体策

   • 「フォルダ全走査/全サムネ生成完了を待たない」ための具体的手順（ストリーミング列挙、最初の候補決定、P0即デコード表示、残りはバックグラウンド）
